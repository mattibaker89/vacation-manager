Vacation Manager

A full-stack vacation request management application built with Vue 3, Node.js, and PostgreSQL.

Vue 3 TypeScript Node.js + Express TypeORM PostgreSQL 16 Vitest + Jest GitHub Actions CI
Features

✈️ Requester Interface

Submit vacation requests with start/end dates and an optional reason. Includes a single-day shortcut checkbox and live status tracking.

📊 Analytics Dashboard

Stat cards for total, pending, approved and rejected requests. Bar, doughnut, and line charts showing trends across the last 6 months.

✅ Request Management

Review all requests at a glance. Filter by status, approve in one click, or reject via a modal that requires a comment before submitting.

👤 User Management

View all users with their roles. Add new Requesters or Validators directly from the UI via a modal form.

🔒 Conflict Protection

The API enforces that only Pending requests can be actioned, preventing duplicate approvals or rejections.

📱 Responsive UI

Bootstrap 5 layout with an Inspinia-inspired admin theme. Tables collapse to cards on mobile screens.

🧪 Full Test Suite

Backend: Supertest integration tests against a real PostgreSQL database. Frontend: Vitest component tests with Vue Test Utils.

⚙️ CI Pipeline

GitHub Actions runs all tests on every push, spinning up a PostgreSQL service container for the backend suite.

Setup & Run
1

Prerequisites

Node.js ≥ 20, pnpm ≥ 9, PostgreSQL 16 running locally.

npm install -g pnpm
2

Clone & install

git clone https://github.com/mattibaker89/vacation-manager.git
cd vacation-manager
pnpm install
3

Configure the environment

macOS / Linux:

cp backend/.env.example backend/.env

Windows:

copy backend\.env.example backend\.env

Then edit backend/.env:

DB_HOST=localhost
DB_PORT=5432
DB_USERNAME=<your_postgres_username>
DB_PASSWORD=<your_postgres_password>
DB_NAME=vacation_manager
PORT=3000

DB_USERNAME: macOS (Homebrew) → your system username (whoami). Windows / Linux → typically postgres.
DB_PASSWORD: macOS (Homebrew) → usually blank. Windows → password set during PostgreSQL installation. Linux → depends on setup.
If running tests, apply the same credentials to backend/.env.test.

4

Start the servers

Open two terminals:

# Terminal 1 — backend on :3000
pnpm dev:backend

# Terminal 2 — frontend on :5173
pnpm dev:frontend

Then open http://localhost:5173 in your browser. The database and seed users are created automatically on first startup.

5

Run the tests

pnpm test

API Reference
MethodEndpointDescription
GET /api/users List all users
POST /api/users Create a new user. Body: { name, role }
POST /api/requests Submit a vacation request
GET /api/requests Get all requests. Optional query params: ?userId= (requester view), ?status= (filter)
PATCH /api/requests/:id Approve or reject a request. Body: { status, comments? }. Comment required when rejecting.
Technical Decisions

Vite over Vue CLI

Vue CLI is in maintenance mode. Vite is the officially recommended build tool for Vue 3 — faster dev server startup and build times via native ESM.

TypeORM synchronize: true

Schema is auto-synced from entities on startup for development convenience. In production this would be replaced with explicit TypeORM migrations. The database itself is also created automatically on first run.

pnpm Workspace

Both backend/ and frontend/ are managed from the root, allowing shared scripts (pnpm test, pnpm dev:backend) without duplication.

Bootstrap 5 UI

Chosen for its responsive grid, pre-built components (modals, badges, tables), and close match to the Inspinia admin theme referenced in the spec.

Supertest Integration Tests

Backend tests hit a real test database rather than mocking TypeORM. This ensures the full request path (validation → ORM → SQL) is exercised.

TypeScript End-to-End

Both backend and frontend are fully typed. Shared type conventions (e.g. RequestStatus, UserRole) keep the API contract explicit and catch mismatches at compile time.

Known Limitations
  • No authentication — user identity is selected from a dropdown for demonstration. A real app would use JWT or session-based auth.
  • synchronize: true is development-only — should be replaced with TypeORM migrations before any production deployment.
  • No pagination — the All Requests page loads all requests at once. For large datasets, server-side pagination would be required.
